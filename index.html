Introduction

Version control is essential for managing code changes, collaborating effectively, and maintaining a clean project history. Git, a distributed version control system, enables developers to track changes, revert to previous states, and work in teams efficiently. This guide covers best practices for using Git effectively.

Setting Up a Git Repository

  1. Install Git (if not already installed) from git-scm.com.

  2. Initialize a new repository in a project folder:

    git init

  3. Clone an existing repository from a remote source:

    git clone <repository_url>

  4. Set up user identity to track commits properly:

    git config --global user.name "Your Name"
    git config --global user.email "your.email@example.com"

  5. Use a .gitignore file to prevent tracking unnecessary files, such as:

    node_modules/
    *.log
    .DS_Store
Writing and Managing Commits

Commit frequently to keep changes small and manageable.

Write clear, concise commit messages. Example:

git commit -m "Fix login issue by updating authentication method"

Stage only relevant changes before committing:

git add <file1> <file2>
git commit -m "Improve error handling in API calls"

Use git commit --amend to fix the last commit (if not pushed yet).

git commit --amend
Branching and Merging Strategy

Always create a new branch for features or bug fixes:

git checkout -b feature/new-feature

Merge changes using pull requests (PRs) to ensure review before merging.

Delete old branches after merging to keep the repository clean:

git branch -d feature/old-branch

Use git pull --rebase instead of merge when appropriate to keep history clean.

git pull --rebase
Handling Syncing and Conflicts

Pull changes before making new commits to avoid merge issues:

git pull origin main

Fetch remote changes without merging immediately:

git fetch origin

Resolve conflicts manually, then mark files as resolved:

git add resolved_file.txt
git commit -m "Resolve merge conflict"
Undoing Changes and Reverting Mistakes

Undo the last commit without losing changes:

git reset --soft HEAD~1

Revert a pushed commit safely:

git revert <commit_hash>

Discard unstaged changes:

git checkout -- filename

Tag releases to mark stable versions:

git tag -a v1.0 -m "First stable release"
git push origin v1.0
Conclusion

Following these Git best practices ensures a smooth workflow, making collaboration easier and keeping your project history clean. Stick to meaningful commits, use branches wisely, and handle conflicts properly.
